# Smartly Bridge Migration Progress

> 日期：2026-07-02
> 分支：`dev`  
> 目標：將 `main` 上的既有 entity-based 架構逐步遷移到六邊形架構與 logical-device / capability-based contract。  
> 原則：所有 runtime 行為變更以 TDD slice 推進；剩餘 cleanup 不再以 legacy fallback 相容為 gate。

## Current Status

- `dev` 已建立 application/domain/adapter/view 分層，並以 ports 讓主要 sync、control、event use case 可以脫離 Home Assistant runtime 測試。
- Devcontainer 已升到 Python 3.14 bookworm image，以符合 Home Assistant 2026.6 `>=3.14.2` 的 test/runtime 依賴要求。
- Adapter manifest validation 已在 application layer 落地，會檢查 stable adapter id、adapter type、canonical capability 宣告、contract_versions、最小 permissions，並偵測 overlapping adapter type/source/domain scope 的 match priority collision；normalization snapshot validation 也會確認 logical-device snapshot 只輸出 manifest 宣告的 capability、source 與 domain；command mapping snapshot validation 會檢查 CommandResult adapter trace 與 expected_state capability scope；event mapping snapshot validation 會檢查 source_event_id / dedupe_key replay-window uniqueness；health snapshot validation 會檢查 status、degraded capability scope 與 unavailable last_error，作為新增 adapter/profile 前的 contract gate。
- Home Assistant `async_setup_entry` 現在透過 `_build_runtime_adapters` composition helper 建立 `runtime_adapters`，將 device event publisher、event deduplicator、local automation rule store、SmartlyCommand executor 與主要 gateway/use case 以六邊形 ports 暴露給 legacy views；device-event endpoint 透過 `_device_event_publisher` / `_device_event_deduplicator` 與 local automation runtime check 要求 setup-created adapters，避免 request-time inline adapter 建立與 runtime 分裂。
